# Plan for Implementing Order Resource

## Goal
Implement an Order resource with all needed methods similar to how Invoice or Booking resources are implemented in the n8n community node.

## Analysis of Existing Resources

### Booking Resource Pattern
From examining `/nodes/RoappRoapp/resources/bookings/index.ts`:
- Operations: getAll, get, create, update, delete, getItems, createItem, updateItem, deleteItem
- Uses shared methods: handleGetAll, handleGetOne, handleCreateUpdate
- BASE_URL constant from shared/methods
- Display options with resource-specific showing

### Invoice Resource Pattern
From examining `/nodes/RoappRoapp/resources/invoices/index.ts`:
- Operations: getAll, get, delete, getItems, create, update, updateStatus, createComment
- Similar structure to bookings but with different operations

## Order API Endpoints Analysis
From ro-app-mcp list-endpoints, the following Order-related endpoints are available:

Core Order Operations:
- GET /orders - Get Orders (already exists)
- POST /orders - Create Order  (already exists)
- GET /orders/{order_id} - Get Order by ID (already exists)
- PATCH /orders/{order_id} - Update Order (already exists)

Order Items:
- GET /orders/{order_id}/items - Get Order Items
- POST /orders/{order_id}/items - Add Order Item
- PATCH /orders/{order_id}/items/{item_id} - Update Order Item
- (DELETE inferred but not explicitly shown)

Additional Order Operations:
- POST /orders/{order_id}/status - Update Order Status
- POST /orders/{order_id}/comments - Create Order Comment
- GET /orders/{order_id}/public-url - Get Order Public Page Url
- GET /orders/statuses - Get Order Statuses
- GET /orders/types - Get Order Types
- GET /orders/custom-fields - Get Order Custom Fields

## Implementation Plan

### 1. Directory Structure
Create `/nodes/RoappRoapp/resources/order/` directory if not exist with the following files (check what is already exist):
- index.ts (main resource file)
- getAll.ts
- get.ts
- create.ts
- update.ts
- delete.ts
- getItems.ts
- createItem.ts
- updateItem.ts
- deleteItem.ts
- updateStatus.ts
- createComment.ts
- getPublicUrl.ts
- getStatuses.ts
- getTypes.ts
- getCustomFields.ts

### 2. File Contents Pattern
Each operation file will follow the pattern seen in bookings/invoices:
- Export a description constant (e.g., `export const ordersGetAllDescription: INodeProperties[] = [...]`)
- The description defines the UI properties for that operation

### 3. Index File Structure (`index.ts`)
- Import all operation description files
- Define display options for order resource: `{ resource: ['order'] }`
- Export `ordersDescription` array combining:
  - Operation selector dropdown
  - All operation description arrays spread in
- Export `executeOrderOperation` function handling all operations

### 4. Operation Implementation
Following the pattern from `executeBookingOperation`:
- getAll: handleGetAll with `${BASE_URL}v2/orders`
- get: handleGetOne with `${BASE_URL}v2/orders/${Id}`
- delete: httpRequestWithAuthentication DELETE method
- getItems: handleGetOne with `${BASE_URL}v2/orders/${Id}/items`
- create: handleCreateUpdate with POST to `${BASE_URL}v2/orders`
- update: handleCreateUpdate with PATCH to `${BASE_URL}v2/orders/${Id}`
- createItem: handleCreateUpdate with POST to `${BASE_URL}v2/orders/${bookingId}/items` (note: uses bookingId parameter like bookings)
- updateItem: handleCreateUpdate with PATCH to `${BASE_URL}v2/orders/${bookingId}/items/${itemId}`
- deleteItem: handleCreateUpdate with DELETE to `${BASE_URL}v2/orders/${bookingId}/items/${itemId}`
- updateStatus: handleCreateUpdate with POST to `${BASE_URL}v2/orders/${Id}/status`
- createComment: handleCreateUpdate with POST to `${BASE_URL}v2/orders/${Id}/comments`
- getPublicUrl: handleGetOne with `${BASE_URL}v2/orders/${Id}/public-url`
- getStatuses: handleGetAll with `${BASE_URL}v2/orders/statuses`
- getTypes: handleGetAll with `${BASE_URL}v2/orders/types`
- getCustomFields: handleGetAll with `${BASE_URL}v2/orders/custom-fields`

### 5. Shared Methods Usage
Import from `'../../shared/methods'`:
- handleGetAll
- handleGetOne  
- BASE_URL
- handleCreateUpdate

### 6. Parameter Naming
Consistent with existing resources:
- Use 'Id' for main resource ID parameters
- For sub-resources like items, use specific ID parameters (e.g., 'itemId')
- Follow the pattern seen in bookings where createItem/updateItem/deleteItem use 'bookingId' and 'itemId'
