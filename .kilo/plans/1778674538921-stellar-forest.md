# Deduplication Agent Plan

## Overview
Add smart deduplication functionality to the existing Roapp n8n node as new operations within the node (triggered from n8n UI). The feature targets **persons** and **organizations** separately (no cross-type matching).

## Feature Breakdown

### 1. New Resource: `deduplication`
Add a new resource option in `RoappRoapp.node.ts` with two operations:

- **`findDuplicates`** — Scan persons and organizations, detect potential duplicates, return a merge report.
- **`merge`** — Execute a merge between two records of the same type based on user selections from the report.

### 2. Matching Criteria & Normalization

#### Normalization
- **Email**: lowercase, trim whitespace
- **Phone**: convert to E.164 format if possible; otherwise normalize by removing non-digit characters (keep leading `+`), trim
- **Name**: trim whitespace, case-fold to lowercase

#### Matching Rules (applied per-type: persons vs persons, organizations vs organizations)
1. **Exact match on normalized email** → duplicate
2. **Exact match on normalized phone** → duplicate
3. **Name similarity** using **Jaro-Winkler** distance with threshold ≥ `0.85` → duplicate

Records are grouped into clusters if any pair within the cluster matches by at least one rule.

### 3. Report Output (`findDuplicates`)

Return an array of duplicate groups:

```json
[
  {
    "cluster_id": "group_1",
    "resource_type": "person", // or "organization"
    "records": [
      { "id": 123, "email": "...", "phone": [...], "name": "...", "created_at": "..." },
      { "id": 456, ... }
    ],
    "match_reasons": [
      { "type": "email", "value": "user@example.com" },
      { "type": "phone", "value": "+1234567890" },
      { "type": "name", "score": 0.92, "compared_to": 123 }
    ],
    "suggested_primary_id": 123 // oldest record (lowest created_at timestamp)
  }
]
```

Additional optional parameters for `findDuplicates`:
- `minNameSimilarity` (number, default: 0.85)
- `limit` (max records to process per type, for large datasets)
- `returnAll` (boolean, fetch all pages)

### 4. Merge Operation (`merge`)

**Input parameters:**
- `resource_type`: `"person"` or `"organization"`
- `record_a_id`: ID of first record
- `record_b_id`: ID of second record
- `merge_direction`: `"auto"` (use suggested_primary from report) or `"manual"` with `primary_id` field
- `conflict_resolution` (optional overrides): map of field → strategy (`"keep_primary"`, `"keep_secondary"`, `"append"`, `"note"`)

**Merge behavior (defaults):**
- **Primary record**: oldest (lowest `created_at`) unless overridden
- **Phones**: append all unique phone numbers from secondary to primary's `phones` array
- **Email**: if primary and secondary have different emails — move secondary email to `notes` field (prepend with "Merged email: ...")
- **Other fields** (name, addresses, custom fields): if different, move secondary's value to `notes` (prepend with field name)
- **Delete** secondary record after merge via API `DELETE` or `PUT` with merge instruction if API supports it

**Output:**
```json
{
  "merged_into_id": 123,
  "merged_record_id": 456,
  "actions_taken": [
    { "field": "phones", "action": "appended", "count": 2 },
    { "field": "email", "action": "moved_to_notes" },
    { "field": "custom_field_x", "action": "moved_to_notes" }
  ],
  "notes_added": "Merged email: ...\nMerged name: ..."
}
```

### 5. User Decision Flow

Two-step process:

**Step 1 – Run `findDuplicates`**
- User runs operation in n8n UI.
- Output shows all duplicate groups with `suggested_primary_id`.
- User reviews and selects which pairs to merge (e.g., via n8n's item filtering or by passing specific cluster IDs to next step).

**Step 2 – Run `merge`**
- User provides list of `(resource_type, record_a_id, record_b_id)` pairs to merge.
- Optionally override primary selection per pair.
- Node executes merges sequentially, returns per-merge result.

### 6. Implementation Details

#### Files to Create/Modify
1. **`@roapp/n8n-nodes-roapp/nodes/RoappRoapp/resources/deduplication/index.ts`** — new resource definition with parameters for both operations.
2. **`@roapp/n8n-nodes-roapp/nodes/RoappRoapp/resources/deduplication/findDuplicates.ts`** — `INodeProperties` description for findDuplicates options.
3. **`@roapp/n8n-nodes-roapp/nodes/RoappRoapp/resources/deduplication/merge.ts`** — `INodeProperties` description for merge inputs.
4. **`@roapp/n8n-nodes-roapp/nodes/RoappRoapp/resources/deduplication/execute.ts`** (or `deduplication.operations.ts`) — execution logic for both operations.
5. **Update `RoappRoapp.node.ts`**:
   - Add `'deduplication'` to resource options
   - Spread `deduplicationDescription` into properties
   - Add case `'deduplication'` in execute switch calling `executeDeduplicationOperation`

#### Helper Functions (modify `shared/methods.ts` or new file)
- `normalizePhone(phone: string): string` → E.164 or digits-only normalized
- `normalizeEmail(email: string): string` → lowercase trim
- `normalizeName(name: string): string` → lowercase, trim, collapse whitespace
- `jaroWinklerSimilarity(s1: string, s2: string): number` (implement or import small lib; verify license)
- `findDuplicates(records: Array<any>, options: any): DuplicateGroup[]`
- `mergeRecords(primary: any, secondary: any, options?: any): MergeResult`

#### API Considerations
- `person` GET: `${BASE_URL}v2/contacts/people`
- `organization` GET: `${BASE_URL}v2/contacts/organizations`
- DELETE endpoint: check API docs — likely `${BASE_URL}v2/contacts/people/{id}` (DELETE) or organizations equivalent. If delete not available, use PATCH to mark as merged/status change and exclude from future runs.

### 7. Success Criteria
- [ ] Deduplication resource appears in n8n node with two operations
- [ ] `findDuplicates` returns groups with cluster_id, record IDs, match reasons, similarity scores
- [ ] Matching uses normalized email/phone and Jaro-Winkler name similarity
- [ ] No cross-type matching (persons vs organizations separate)
- [ ] `merge` operation correctly merges two records, handling conflicts per rules
- [ ] Phones array merged; conflicting email/other fields moved to notes
- [ ] Secondary record deleted/marked after merge
- [ ] Lint and typecheck pass

### 8. Questions / Open Points
- Does the Roapp API support direct deletion of persons/organizations? If not, what field can we update to mark as merged (e.g., `status_id`, custom field)? We need to check API docs or test.
- Should `findDuplicates` support incremental runs (only new records since last run)? Not in v1 — full scan each time.
- Should there be a threshold for Jaro-Winkler as a parameter or fixed? Plan: parameterized with default 0.85.
- Should merge be atomic (all-or-nothing per pair) or continue on error? Plan: stop on error, report failed merges.
